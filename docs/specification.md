## Демон для мониторинга системных ресурсов с клиентом

**Актуальность**: потребность отслеживания состояния серверов, использование данных для оптимизации распределения нагрузки.

**Суть разработки**:
Демон (сервис), который собирает метрики (загрузка CPU, использование памяти, диска, температура CPU) с заданным интервалом. Клиент, который обрабатывает пользовательские команды и возвращает запрошенные данные.

  

**Функциональные требования демона**:
- Сервис устанавливается на сервер и мониторит состояние только этого сервера.
- Требования к режиму работы:
	- Запускается автоматически при старте операционной системы;
	- Работает в фоновом режиме как системная служба;
	- Функционирует непрерывно до завершения работы или перезагрузки системы.
- Отслеживаемые метрики:
	- Загрузка CPU (сколько времени процессор работает, простаивает);
	- Использование памяти (сколько ОЗУ занято, свободно);
	- Использование диска (заполненность разделов, скорость чтения/записи);
	- Температура CPU.
- Структура записи метрик: помимо полей с метриками сохраняется поле timestamp, содержащее информацию о моменте времени снятия данных в формате Unix timestamp.
  - Пример записи снятия метрик:

```json
{
  “timestamp”: 2026-01-15T1010:30:00Z”,
  “server”: “server-a”,
  “cpu_usage”: 19.9,
  “memory_usage”: 63,
  “disk_usage”: 87.2,
  “temperature”: 61
}
```

- Интервал сбора метрик: указывается в конфигурационном файле. Ограничения сверху и снизу на значение интервала определяются в процессе разработки.
- Отправка данных: собираемые данные отправляются на сторонний сервер. Его URL-адрес и endpoint указывается в конфигурационном файле.
- Предусмотрена повторная отправка данных через некоторый промежуток времени. Промежуток для повторной отправки указывается в конфигурационном файле.
- Алгоритм передачи данных:
	- Буферизация меток с последующей пакетной отправкой.
	- Периодичность отправки указывается в конфигурационном файле в диапазоне 1–300 секунд.
	- Поддержка отправки при значительном изменении метрик. Порог изменения и минимальный интервал между отправками указываются в файле конфигурации.
- Конфигурация: при запуске определяются начальные настройки сервиса в соответствии с файлом конфигурации. Могут быть добавлены дополнительные поля в процессе реализации.

Пример конфигурационного файла:
```json
{
  “monitor_interval”: 0,1,
  “repeat_interval”: 10,
  “send_interval”: 5,
  “min_send_interval”: 1,
  “send_strategy”: [
      “url”: “https://...”,
      “endpoint”: “/server-a/metrics”
  ],
  “max_packet_size”: 1Mb
}
```

**Функциональные требования клиента**:
- Предоставляет пользовательский интерфейс: консольное приложение с системой команд для выгрузки метрик в файл. Конкретный формат команд задается в процессе разработки.
- Обработка команд пользователя: клиент запрашивает данные из сервера хранения. Набор отслеживаемых серверов указывается в конфигурационном файле. Поле содержит словарь с полями адресов отслеживаемых серверов и соответствующих им endpoint-ов на сервере хранения.
- Предупреждения: клиент сравнивает полученные данные и соответствующие критические значения, в случае превышения критических значений вместе  с ответом отправляет пользователю предупреждение. Критические значения указываются в конфигурационном файле в поле.
- Возвращение полученных данных:
	- Сохранение в файл. Имя файла указывается командой пользователя.
	- Сохранение в базе данных: параметры для подключения указываются в конфигурационном файле.
	- Формат, в котором сохраняются данные, определяется в процессе разработки.
- Конфигурация: при запуске определяются начальные настройки клиента в соответствии с файлом конфигурации. Могут быть добавлены дополнительные поля в процессе реализации.

Пример конфигурационного файла:

```json
{
  “servers”: [
      “server-a”: “/server-a/metrics”,
      “server-b”: “/server-b/metrics”
  ],
  “critical”: [
    “cpu_usage”: 80,
    “memory_usage”: 85,
    “disk-usage”: 90,
    “temperature”: 90
  ],
  “bd_params”: […]
}
```
